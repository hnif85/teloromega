   128→   128→## PHASE 0: AUTH — SSO mwxmarket.ai
   129→   129→
   130→   130→> **Tidak ada form register/login terpisah.** Semua auth memakai sistem existing mwxmarket.ai.
   131→   131→
   132→   132→### Step 0.1 — Login Flow
   133→   133→
   134→   134→```javascript
   135→   135→USER                                  NEXT WHIZ                    mwxmarket.ai
   136→   136→────                                  ─────────                    ────────────
   137→   137→
   138→   138→1. Buka nextwhiz.id
   139→   139→                              ──▶   2. Cek session (JWT cookie)
   140→   140→                                      ├── Valid? YA → Dashboard
   141→   141→                                      └── TIDAK → redirect ▼
   142→   142→                              ──▶   3. Redirect ke:
   143→   143→                                      mwxmarket.ai/auth/login
   144→   144→                                      ?redirect_uri=nextwhiz.id/callback
   145→   145→                                                             ──▶  4. User login/register
   146→   146→                                                             ◀──  5. Return JWT + user data
   147→   147→                              ◀──   6. Terima callback:
   148→   148→                                      • Simpan JWT di cookie
   149→   149→                                      • Extract: user_id, email, name
   150→   150→                                      • UPSERT user + cache credit
   151→   151→                                      ▼
   152→   152→                              ──▶   7. Redirect:
   153→   153→                                      • First time? → Onboarding (Phase 1)
   154→   154→                                      • Returning? → Dashboard (Phase 4)
   155→   155→```
   156→   156→
   157→   157→### Step 0.2 — Data yang Diterima dari mwxmarket.ai
   158→   158→
   159→   159→| Field            | Source           | Dipakai untuk                           |
   160→   160→| ---------------- | ---------------- | --------------------------------------- |
   161→   161→| `user_id`        | JWT claim        | Primary key user di Next Whiz           |
   162→   162→| `email`          | JWT claim        | Identitas user                          |
   163→   163→| `name`           | JWT claim        | Nama tampilan di sidebar & dashboard    |
   164→   164→| `credit_balance` | API mwxmarket.ai | Saldo credit (dipakai untuk semua aksi) |
   165→   165→
   166→   166→> **Catatan:** tidak ada `subscription_tier`. Gating fitur sepenuhnya berbasis saldo credit — kalau credit cukup, aksi jalan; kalau tidak, modal top-up.
   167→   167→
   168→   168→### Step 0.3 — Sync User ke Next Whiz DB
   169→   169→
   170→   170→| Aspek               | Detail                                                                             |
   171→   171→| ------------------- | ---------------------------------------------------------------------------------- |
   172→   172→| **Kapan**           | Setiap kali user login via SSO callback                                            |
   173→   173→| **System action**   | `UPSERT` ke `shared.users`. Update `credit_balance` (cache), `name`, `last_login`. |
   174→   174→| **First-time user** | `INSERT` user row. Belum ada brand → redirect onboarding (Phase 1).                |
   175→   175→| **Data minimal**    | `{ user_id, email, name, credit_balance, last_login }`                             |

   196→   196→## PHASE 2: PRODUK — Barang atau Jasa

   202→   202→
   203→   203→| Aspek         | Detail                                                                                    |
   204→   204→| ------------- | ----------------------------------------------------------------------------------------- |
   205→   205→| **User sees** | 📦 **Barang** — "Saya jualan produk fisik" · 💼 **Jasa** — "Saya menawarkan jasa/layanan" |

   213→   213→| Harga Jual (Rp)      | ✅      | 15.000                                                                             |

   233→   233→
   234→   234→### Step 2.4 — Data Model Produk
   235→   235→
   236→   236→```sql
   237→   237→shared.products (
   238→   238→  id UUID PK,
   239→   239→  brand_id UUID FK,
   240→   240→  type TEXT CHECK(type IN ('barang','jasa')),  -- ★ pembeda
   241→   241→  name TEXT NOT NULL,
   242→   242→  price INT NOT NULL,        -- harga jual
   243→   243→  cost_price INT NULL,       -- ★ harga modal → margin/laba otomatis
   244→   244→  stock INT NULL,            -- ★ SUMBER STOK OTORITATIF (barang). NULL untuk jasa
   245→   245→  min_stock INT NULL,
   246→   246→  sku TEXT NULL,
   247→   247→  description TEXT,
   248→   248→  image_url TEXT,
   249→   249→  is_active BOOLEAN DEFAULT true,
   250→   250→  created_at TIMESTAMP
   251→   251→)
   252→   252→
   253→   253→shared.inventory (           -- ★ HANYA kalau produk punya varian
   254→   254→  id UUID PK,                -- (mis. ukuran/rasa). Tanpa varian, tabel ini
   255→   255→  product_id UUID FK,        -- tidak dipakai; products.stock yang berlaku.
   256→   256→  variant TEXT NOT NULL,
   257→   257→  stock INT,
   258→   258→  updated_at TIMESTAMP
   259→   259→)
   260→   260→```
   261→   261→
   262→   262→> **Aturan stok (satu sumber):** produk tanpa varian → `products.stock` satu-satunya sumber. Produk dengan varian → `products.stock` = SUM(`inventory.stock`) (derived, di-recompute tiap perubahan varian). Tidak pernah ada dua angka stok yang di-update independen.
   263→   263→

   274→   274→## PHASE 3: RISET PERTAMA (Disarankan — Bisa Skip)
   275→   275→
   276→   276→> Setelah brand + produk (kalau ada), sistem menyarankan riset pertama.\
   277→   277→> Riset **tidak** dianalisis per produk user (masih MVP), tapi **query bisa di-pre-fill dari nama produk**.
   278→   278→
   279→   279→### Step 3.1 — Riset Suggestion Screen
   280→   280→
   281→   281→| Aspek                 | Detail                                                                                                                                                                                                                                                                      |
   282→   282→| --------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
   283→   283→| **User sees**         | Card: "✨ Yuk riset pasar dulu! Biar tau tren, kompetitor, dan peluang buat [nama brand] kamu."                                                                                                                                                                              |
   284→   284→| **Suggested queries** | Kalau **ada produk** → pre-fill pakai nama produk: "Tren [nama produk]", "Harga pasaran [nama produk]", "Kompetitor [nama produk]". Kalau **belum ada produk** → fallback ke kategori brand: "Tren [kategori] 2026", "Keyword trending [kategori]". Plus search bar manual. |
   285→   285→| **User action**       | Klik suggestion / ketik manual → "Riset"                                                                                                                                                                                                                                    |
   286→   286→
   287→   287→### Step 3.2 — Pipeline Riset
   288→   288→
   289→   289→| Step                           | System Action                                                                                                                                                                                                 |
   290→   290→| ------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
   291→   291→| **1. Cek credit**              | `credit_balance >= credit_rates['riset.pasar']`? Kalau tidak → modal top-up (lihat Phase 5).                                                                                                                  |
   292→   292→| **2. Klasifikasi intent**      | LLM: `market_trend` / `competitor_analysis` / `keyword_research` / `pricing`                                                                                                                                  |
   293→   293→| **3. Kumpulkan data real**     | **Parallel:** Google Trends (pytrends) + Tavily (web + social) + marketplace scrape (Shopee/Tokopedia)                                                                                                        |
   294→   294→| **4. Kirim ke LLM**            | Prompt: data real + konteks brand (kategori, nama, tone) → SWOT, persona, kompetitor, keyword, rekomendasi                                                                                                    |
   295→   295→| **5. Simpan riset**            | `INSERT` ke `shared.research` (`status='completed'`, `result_json`)                                                                                                                                           |
   296→   296→| **6. Potong credit**           | Potong via API mwxmarket + `INSERT` `credit_usage_log` (lihat aturan idempotency Phase 5)                                                                                                                     |
   297→   297→| **7. ★ Auto-generate context** | Context Engine langsung bikin **3 context** (konten, toko, keuangan) dari riset ini → `INSERT` `shared.context` ×3. **Tanpa potong credit** (turunan gratis). Ini yang mengisi Rekomendasi Aksi di dashboard. |

   551→   551→### Step 5.1 — Prinsip Sumber Kebenaran
   552→   552→
   553→   553→| Aspek                 | Aturan                                                                                                                                                                                                |
   554→   554→| --------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
   555→   555→| **Sumber otoritatif** | `credit_balance` di mwxmarket.ai. **Selalu** dicek live via API sebelum aksi.                                                                                                                         |
   556→   556→| **Cache lokal**       | `shared.users.credit_balance` hanya cache untuk tampilan cepat/fallback. Tidak pernah jadi dasar keputusan potong.                                                                                    |
   557→   557→| **Idempotency**       | Tiap aksi punya `reference_id` unik. Potong credit dikirim dengan idempotency key → kalau retry, tidak dobel potong.                                                                                  |
   558→   558→| **Urutan aman**       | (1) cek saldo → (2) potong via API mwxmarket (dapat `balance_after`) → (3) baru jalankan aksi → (4) log. Kalau aksi gagal setelah potong → **refund** via API. Kalau potong gagal → aksi tidak jalan. |

   898→   898→## 📋 EDGE CASES
   899→   899→
   900→   900→| Skenario                                       | Handling                                                                                            |
   901→   901→| ---------------------------------------------- | --------------------------------------------------------------------------------------------------- |
   902→   902→| **SSO token expired**                          | Redirect mwxmarket re-auth. Simpan intended URL, balik setelah login.                               |
   903→   903→| **User belum setup brand**                     | Redirect onboarding Phase 1. Tidak bisa akses modul.                                                |
   904→   904→| **Credit habis di tengah aksi**                | Cek SEBELUM proses (saldo live). Tidak cukup → modal top-up. Tidak ada partial execution.           |
   905→   905→| **Potong credit sukses tapi aksi gagal**       | Refund via API mwxmarket (idempotency key). Log status='refunded'.                                  |
   906→   906→| **Retry aksi yang sama**                       | Idempotency key (`reference_id`) → tidak dobel potong.                                              |
   907→   907→| **Research gagal / timeout**                   | Error state + retry. Credit di-refund. Context tidak dibuat.                                        |
   908→   908→| **Produk jasa**                                | Tidak masuk inventory, tidak masuk alur shipping. Order jasa: `resi/courier/shipping_cost = NULL`.  |
   909→   909→| **Barang tanpa cost_price**                    | Margin/laba ditandai "belum lengkap" + prompt isi modal. Income tetap tercatat (tanpa HPP).         |
   910→   910→| **Barang tanpa foto**                          | Placeholder generik (inisial nama).                                                                 |
   911→   911→| **Stok bervarian**                             | `products.stock` = SUM(inventory.stock), di-recompute tiap perubahan. Bukan dua angka independen.   |
   912→   912→| **Slug brand bentrok**                         | Auto-suffix (`keripik-ani-2`).                                                                      |
   913→   913→| **Order tapi stok kurang**                     | Validasi sebelum order. Kalau kurang → warning + opsi pre-order.                                    |
   914→   914→| **Pembayaran tidak diverifikasi > 2 hari**     | Reminder dashboard. > 7 hari → auto-cancel order + stok balik. Income belum pernah tercatat (aman). |
   915→   915→| **Context dipakai berkali-kali**               | Diizinkan. Tiap pemakaian → `context_usage`. Context tidak dihapus.                                 |
   916→   916→| **Brand dihapus**                              | Soft delete (`is_active=false`). Data historis tetap.                                               |
   917→   917→| **User switch brand saat proses jalan**        | Modal konfirmasi. Proses tidak dibatalkan; UI reload brand baru.                                    |
   918→   918→| **Transaksi tanpa produk (biaya operasional)** | `product_id` nullable, `category` wajib.                                                            |
   919→   919→| **Customer duplikat (nomor sama)**             | Match by phone per brand → update customer existing, bukan bikin baru.                              |
   920→   920→| **Piutang/hutang jatuh tempo**                 | Cek harian. ≤ 3 hari → notifikasi dashboard.                                                        |
